01 — Visual Theme & Atmosphere

Warm paper,
not sterile glass.

Notion's website embodies the philosophy of the tool itself — a blank canvas that gets out of your way. The page is pure white, but the text isn't pure black; it's a warm near-black. The grays carry yellow-brown undertones, giving the interface a tactile, almost analog warmth.

Paper-like

Pure white (#ffffff) canvas with warm near-black text at rgba(0,0,0,.95). The 5% transparency softens black without sacrificing readability.

Whisper borders

Rather than heavy borders or shadows, Notion uses ultra-thin 1px solid rgba(0,0,0,0.1) borders — barely perceptible division lines that create structure without weight.

Layered shadows

Multi-layer shadow stacks with cumulative opacity never exceeding 0.05 — depth that is felt rather than seen. Natural light, not computer-generated drop shadow.

Compressed display type

Inter at display (64px) uses -2.125px letter-spacing. Compression creates density at headlines while body text relaxes to neutral tracking. (Notion's private NotionInter is a modified Inter — we approximate it via OpenType features.)

04 — Typography Hierarchy

Sixteen roles,
four weights,
one family.

Compression tightens as size grows. −2.125px at 64px, progressively relaxing to normal at 16px. Four weights: 400 (read), 500 (interact), 600 (emphasize), 700 (announce).

Font note

Notion ships a private modified variant of Inter called NotionInter — not publicly available. This board loads Inter (via Google Fonts) with OpenType features ss01 (single-storey a), cv11 (narrower digits), lnum, and locl enabled to approximate NotionInter's tighter aperture and compressed display tracking. Expect ~5–8% visual drift vs. notion.so — most visible at 64px+. Anywhere the spec says NotionInter, use font-family: 'Inter' in production.

05 — Typography Principles

Compression scales
with size.

The bigger the type, the tighter it gets. Four rules govern the entire system.

Compression at scale

64px → −2.125px · 54px → −1.875px · 48px → −1.5px · 26px → −0.625px · 16px → normal. Compression creates density at headlines while body text stays legible.

Four-weight system

400 reading · 500 interactive · 600 emphasis / navigation · 700 headings / display. Broader weight range than most systems for nuanced hierarchy.

Warm scaling

Line height tightens as size grows: 1.50 at body → 1.23–1.27 at sub-heading → 1.00–1.04 at display. Denser headlines, breathing body copy.

Badge micro-tracking

The 12px badge is the only positive-tracked text in the system (+0.125px). Creates wider, more legible small text where density would otherwise kill readability.

18 — Iteration Guide

Eight rules you
can't violate.

Print this list and tape it to your monitor. If you break any of these, it stops being Notion.

Always use warm neutrals. Notion's grays have yellow-brown undertones (#f6f5f4, #31302e, #615d59, #a39e98) — never blue-gray.
Letter-spacing scales with font size: −2.125px at 64px, −1.875px at 54px, −0.625px at 26px, normal at 16px.
Four weights only: 400 (read), 500 (interact), 600 (emphasize), 700 (announce).
Borders are whispers: 1px solid rgba(0,0,0,0.1) — never heavier.
Shadows use 4–5 layers with individual opacity never exceeding 0.05.
The warm white (#f6f5f4) section background is essential for visual rhythm. Alternate white ↔ warm white.
Pill badges (9999px) for status/tags, 4px radius for buttons and inputs.
Notion Blue (#0075de) is the only saturated color in core UI — use it sparingly for CTAs and links.

Rules of the name

Semantic over visual. Use --color-text-primary, not --color-black-95. Visual names break when the brand shifts; semantic names survive.
State is a suffix, not a prefix. --color-accent-hover (good). --hover-color-accent (bad).
No component names in base tokens. --button-bg is a component token. It should reference a base: --button-bg: var(--color-accent). Two-layer system — never hardcode.
Scale uses numbers, role uses words. Spacing/radius follow --space-8. Semantic roles follow --color-text-secondary. Don't mix.

Why four surfaces per state?

Reusing the same hex everywhere (e.g., "just use #1aae39 for success") breaks WCAG contrast the moment you need success text on a success background. The four-surface family guarantees readable, consistent treatment across alerts, toasts, banners, inline messages, badges, and form validation — without re-deciding each time.

Three curves.

Without motion tokens, every developer invents a different transition duration. Lock three values upfront — refine later if needed. Rarely needed.

Duration scale

120ms · fast

Micro-interactions

Button hover, icon state, input focus ring. Should feel immediate — no lag, no animation.

200ms · base

Standard transitions

Card hover lift, modal open, tab switch, dropdown reveal. The default for almost everything.

400ms · slow

Content reveal

Page transitions, large panel slides, hero illustration entrances. Anything asking for attention.

Easing curves

cubic-bezier(0.2, 0, 0, 1)

Ease-out

Default for UI. Fast start, soft stop. Feels responsive.

cubic-bezier(0.4, 0, 0.2, 1)

Ease-in-out

Soft both ends. Modal opens, page transitions.

cubic-bezier(0.34, 1.56, 0.64, 1)

Ease-spring

Slight overshoot. Playful micro-delights only. Rare.

Respect reduced-motion

Wrap non-essential motion in @media (prefers-reduced-motion: reduce). Duration → 0ms, transforms → none. Essential feedback (focus rings, button press) stays.

06 · Do & Don't
✓ Do
  • One primary button per view
  • Action-oriented label ("Send package", not "OK")
  • Show loading state while async work runs
  • Use destructive variant for delete / discard
✗ Don't
  • Don't stack two primary buttons side by side
  • Don't use ghost button for the main action on a page
  • Don't disable without an obvious reason ("Why is this gray?")
  • Don't rely on color alone for destructive (use word "Delete")
